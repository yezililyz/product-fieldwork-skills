# Product Vision

Grounded in "Product Strategy — A Framework for Filtering Great from Good" and "Business Strategy — Creating Competitive Advantage."

## When to use
Use this when a "vision" document is actually describing next quarter's roadmap — mentioning specific features or technologies — instead of the 3-5 year destination it's supposed to describe; a real vision almost never mentions features, since those belong in strategy and roadmap layers beneath it. It's also the right tool when a vision sounds inspiring but points toward nothing that would actually be defensible if achieved — checking it against VRIN (valuable, rare, inimitable, non-substitutable) is what separates an aspiration from a real destination worth building toward.

## What it does
This skill separates vision from strategy explicitly using the top-down pyramid (Vision → Principles → Mission → Strategic Plays → Tactics → Metrics), keeping the vision layer free of feature or technology specifics, and pressure-tests the vision's substance with VRIN — does the destination the vision describes actually point toward a capability that would be valuable, rare, hard to imitate, and non-substitutable once reached, or is it just an inspiring sentence with no real moat behind it.

## Method
1. State the vision as the fixed 3-5 year destination only — what does success look like if this fully plays out — with zero mention of specific features, technologies, or the current roadmap; if a vision statement name-drops a specific tool or integration, that content belongs one layer down in strategy, not in vision.
2. Write the vision to answer one question only: if we succeed, what does this become? Not how we'll get there (that's mission/strategy) and not what ships next (that's roadmap/PRD).
3. Check the vision against VRIN as a destination, not just as a sentence: if the organization actually reached this vision, would the resulting position be valuable (exploits a real opportunity), rare (few others would occupy it), inimitable (genuinely hard to copy, not just uncopied so far), and non-substitutable (can't be swapped for a generic alternative)? A vision that fails this test is inspiring but not strategically load-bearing.
4. Keep principles (the core values that filter every idea below the vision) distinct from the vision itself — principles are the sieve; the vision is the destination the sieve protects the path toward.
5. Build the one-page canvas for stakeholder alignment separately from the full vision document: a one-line vision, the customer pain points it addresses, and the two pillars (Foundation — making the core solid; Growth — expanding reach) that connect vision to near-term work without collapsing the distinction between them.
6. Match update cadence to the vision's actual time horizon — a real vision changes rarely (years, only on major strategic shifts), while strategy revisits every 6-12 months and roadmap refreshes quarterly; rewriting the vision every quarter is a sign it was never actually a vision, just a longer-range roadmap.
7. Trace every strategic pillar and roadmap item back up to the vision explicitly — if a roadmap item can't be connected to the vision through strategy, that's the signal the hierarchy was skipped somewhere, not that the roadmap item is necessarily wrong.
8. Revisit the VRIN check periodically even for an established vision — a destination that was genuinely defensible when set can lose that quality if the capability it depends on becomes commoditized industry-wide before the vision is fully reached.

## Inputs
- The current vision draft (or its absence), checked for whether it mentions features/technology that belong one layer down
- The underlying capability or position the vision describes, to run the VRIN check against
- The full pyramid (principles, mission, strategic plays) if it exists, to check the vision traces down cleanly through it
- How frequently the vision has actually been rewritten, as a check against the "years, not quarters" cadence rule

## Output format
A vision statement with zero feature/technology mentions, stating only the 3-5 year destination; a VRIN verdict on the underlying capability the vision points toward; a one-page canvas (vision, customer pain points, Foundation/Growth pillars) for stakeholder alignment; and a stated update cadence appropriate to the vision's actual time horizon (years, not quarters).

## Example
A company's draft vision reads "become the leading AI-powered analytics platform using real-time streaming architecture." Stripped of technology and feature mentions, the real vision is closer to "become the system of record analysts trust for every real-time business decision" — and the VRIN check reveals the real defensible destination isn't the streaming architecture (increasingly commoditized) but the proprietary, compounding dataset that real-time usage would generate over years, which is what the vision should actually be pointing toward.

## Common pitfalls
- Writing a vision that names specific features or technologies, collapsing the vision layer into what's actually a medium-term strategy or roadmap statement.
- Treating an inspiring sentence as sufficient without checking whether the destination it describes would actually be defensible (VRIN) once reached — vision without a real moat is just a slogan.
- Rewriting the vision every quarter alongside the roadmap, which signals it was never functioning as a true multi-year destination in the first place.
